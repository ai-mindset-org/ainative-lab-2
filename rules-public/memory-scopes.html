rule

memory scopes: user · task · project · operational

всё всегда в одном бакете памяти, и всё уплывает — личные предпочтения смешиваются с эфемерным контекстом задачи, и оба теряются.

когда применять

при дизайне любого долгоживущего агента или ассистента. одна память — это анти-паттерн, потому что разные виды памяти живут по разным законам и требуют разной политики удержания.

четыре scope

user memory — про человека, кросс-сессионная, переживает проекты. task memory — про конкретный шаг, эфемерная, удаляется после завершения. project memory — привязана к артефакту, живёт пока живёт проект. operational memory — общая для команды, видна всем участникам.

как применять

у каждого scope свои правила: что попадает, что промотируется в следующий уровень, что архивируется, что удаляется. user memory не должна загрязняться task-фрагментами; project memory не должна тянуть в себя личные привычки.

когда scope явный — память служит. когда всё в одной куче — память мешает: модель тянет старые task-детали в новую задачу и принимает их за актуальные. четыре корзины дешевле одной мутной.
