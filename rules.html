минимальный harness для personal os

13 правил, шесть зон контура. harness — это не инструмент и не фреймворк, это рамка вокруг агентного процесса, в которой ты остаёшься оператором, а не пассажиром. кликни зону — отфильтруешь правила. кликни правило — раскроется тело и ссылка на полную страницу.
fig.1 шесть зон контура click to filter rules
fig.2 что значит «harness вокруг внешней силы» verbal frame

harness — это контур из шести зон вокруг агентного процесса. контекст направляет, память помнит, guardrail ограничивает, review корректирует, naming делает находимым, communication передаёт. без одной из них — рваный harness, утечка ответственности.

агент — это внешняя сила. он делает то, что ты задал, плюс то, что не заметил. harness — шесть стенок коробки, в которой эта сила работает на тебя, а не разносит вокруг. каждое правило закрывает одну щель.

проверь себя: какая зона слабее всего? нет evidence — не помнишь почему. нет blast radius — агент пишет в продакшн. нет naming — ничего не находится. рваный harness ломается там, где тоньше.

r.01 context seed: source · signal · ask · output · evidence context [ expand ]
агент получает «сделай задачу» без рамы и галлюцинирует

каждый запуск задачи начинается с пяти полей. source — откуда данные. signal — что ты заметил. ask — что хочешь получить. output — формат ответа. evidence — что считать доказательством. без этого harness разваливается на первых трёх запросах. пока пять полей не заполнены — не запускай.

r.02 one workflow перед архитектурой context [ expand ]
не знаешь, с чего начать, поэтому строишь всё сразу

выбери одну рабочую задачу, которая повторяется. одна. без подзадач. собери harness под неё — context, шаблоны, инструменты, границы. остальные задачи появятся сами через паттерны. лучше, чем строить «универсальную систему».

r.03 template before dashboard context [ expand ]
тратишь день на дашборд, который никто не использует

сначала шаблон в .md. потом — если шаблон выживает три недели — дашборд. дашборды дороги в обслуживании, шаблоны почти бесплатны. артефакт сначала текстом, потом интерфейс.

r.04 claude context window — окно как ресурс context [ expand ]
контекст переполнен на середине задачи, агент теряет нить

смотри на остаток токенов в status line, не на проценты. ok > 400k, warn 150–400k, crit < 150k. в crit — только handoff, не новые задачи. контекст — ресурс, а не бесконечность; на каждом ответе footer с режимом.

r.05 evidence > logs (decision memory) memory [ expand ]
через неделю не помнишь, почему принял решение

memory — это не лог что произошло, а доказательство почему. сохраняй цитаты, скриншоты, transcripts, ссылки на источники. лог теряется — evidence держит решение воспроизводимым. через месяц ты пересматриваешь, не вспоминаешь.

r.06 memory scopes: user · task · project · operational memory [ expand ]
всё валится в одну заметку, агент не находит нужное

четыре уровня. user — кто я, мои принципы, контакты. task — что делаю прямо сейчас. project — постоянный контекст продукта или лаборатории. operational — как я работаю, привычки, инструменты. без scope memory становится свалкой, агент мешает личное и проектное.

r.07 agent passport (owner · scope · tools · expiry) guardrail [ expand ]
агенты копятся, никто не знает, что они умеют и кем созданы

у каждого агента есть карточка-паспорт. owner — кто отвечает. scope — что разрешено делать. tools — какие инструменты подключены. expiry — когда переаудит. без паспорта агент дрейфует — через полгода двадцать скиллов и ты не помнишь, который пишет в продакшн.

r.08 guardrails как blast radius guardrail [ expand ]
агент написал в продакшн, удалил файлы, отправил сообщения не туда

guardrail = радиус взрыва, который ты готов разрешить. read-only по умолчанию. write только в sandbox. деструктивные действия — через явное подтверждение или отдельный канал. до запуска отвечай: что максимум он сломает? нет ответа — не запускай. это граница между «эксперимент» и «инцидент».

r.09 human review как сигнал, не одобрение review [ expand ]
review превращается в формальность, ничего не меняется

review это не «одобрить или нет». это сигнал, который ты возвращаешь системе. что было неточно? какая часть промпта не сработала? review копится в evidence и формирует следующую итерацию rule. без обратной связи система не учится.

r.10 replay-ability — каждое действие воспроизводимо review [ expand ]
агент сделал что-то крутое один раз, повторить не получается

каждое действие = input + prompt + version + output. сохраняй четыре. через месяц ты должен уметь запустить тот же prompt на тех же inputs и получить близкий output. это не детерминизм — это доверие. систему, которую нельзя проиграть заново, не отличить от удачи.

r.11 auto-eval (T/R/C) на каждый ответ review [ expand ]
не понимаешь, где система теряет качество

на каждом значимом ответе три бинарных оценки. T (text) — структура и ясность. R (rules) — соблюдение naming, форматов, конвенций. C (code) — работает ли. ✓ pass, ✗ fail, ○ n/a. через месяц паттерны fail подсказывают, какое правило добавить. binary, не шкала: «3 из 5» ничего не значит, fail — значит.

r.12 naming как операционная ось naming [ expand ]
агент не находит файлы, vault превращается в свалку

имя файла — api между тобой и агентами. {project} {type} description – yyyy-mm-dd.md. project и type в скобках парсятся одной регуляркой. описание lowercase, дата в конце — ls сортируется хронологически. без convention vault слепнет.

r.13 communication: рамка проектная, без эмоджи, en dash communication [ expand ]
сообщения команде звучат как корпоративный спам

строчная буква в начале блоков. без эмоджи (если только пользователь явно не просит). главная мысль в первых 2–3 строках. конкретно — кто, что, когда, ссылка. en dash (–), не em dash (—). без «в рамках», «в контексте», «извините за беспокойство». проектная рамка, не корпоративная.
